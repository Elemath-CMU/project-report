\section{\ifenglish Challenges\else ปัญหาที่พบและแนวทางการแก้ไข\fi}

ในการทำโครงงานนี้ พบว่าเกิดปัญหาหลัก ๆ ดังนี้
\begin{enumerate}
    \item เกิดปัญหาคอขวด (bottleneck) ในขั้นตอนการออกแบบ สาเหตุมาจากการประเมินปริมาณงานและความซับซ้อนของการออกแบบ UX/UI ต่ำกว่าความเป็นจริงในช่วงเริ่มต้น 
    ทำให้มีผู้รับผิดชอบในส่วนของการออกแบบเพียงคนเดียว ส่งผลให้กระบวนการทำงานในขั้นตอนอื่น ๆ ที่ต้องรอรูปแบบการออกแบบเกิดความล่าช้า 
    ทางผู้จัดทำจึงแก้ไขโดยการเพิ่มจำนวนสมาชิกเข้ามาช่วยรับผิดชอบในส่วนของการออกแบบเพิ่มเติม เพื่อกระจายภาระงานและลดความล่าช้าในขั้นตอนดังกล่าว
    \item การกระจายภาระงานที่ไม่สมดุล สาเหตุมาจากการขาดการแจกแจงรายละเอียดโครงสร้างงาน (task breakdown) ที่ชัดเจนตั้งแต่เริ่มโปรเจกต์ ทำให้ในช่วงแรกของการดำเนินงาน 
    การแบ่งหน้าที่รับผิดชอบของสมาชิกในกลุ่มยังไม่เหมาะสมและไม่สอดคล้องกับปริมาณงานที่เกิดขึ้นจริง ส่งผลให้สมาชิกบางคนมีภาระงานที่หนักเกินไป 
    ทางผู้จัดทำจึงแก้ไขโดยการนำกระบวนการทำงานทั้งหมดมาทบทวนร่วมกันใหม่ และจัดสรรภาระงานให้มีความสมดุลมากขึ้น เพื่อให้การดำเนินงานเป็นไปอย่างมีประสิทธิภาพและเสร็จสิ้นตามกำหนดเวลา
    \item การสื่อสารภายในกลุ่มผู้จัดทำที่ไม่สม่ำเสมอ สาเหตุมาจากเวลาว่างของสมาชิกในกลุ่มที่ไม่ตรงกัน และไม่มีการกำหนดช่องทางหรือความถี่ในการอัปเดตงานร่วมกันอย่างเป็นทางการตั้งแต่แรก 
    ทำให้เกิดความคลาดเคลื่อนในการทำงานและทำความเข้าใจรายละเอียดของงานไม่ตรงกัน ทางผู้จัดทำจึงแก้ไขโดยการเพิ่มความถี่ในการติดตามความคืบหน้าของงานอย่างใกล้ชิด 
    และหมั่นสอบถามสถานะการดำเนินงานในแต่ละส่วนอย่างสม่ำเสมอ เพื่อให้สมาชิกทุกคนเห็นภาพรวมของงานตรงกันและสามารถดำเนินการต่อไปได้ตามแผนที่วางไว้
    \item ารบริหารจัดการเวลาในส่วนของการพัฒนาระบบ สาเหตุมาจากผลกระทบสะสมจากความล่าช้าในขั้นตอนการออกแบบ รวมถึงการประเมินระยะเวลาที่ใช้ในการเขียนโค้ดคลาดเคลื่อน 
    ทำให้ในช่วงแรกของการพัฒนาระบบ เกิดความล่าช้าและมีความเสี่ยงที่จะดำเนินการไม่แล้วเสร็จตามกำหนดเวลา ทางผู้จัดทำจึงแก้ไขปัญหาโดยการนำส่วนประกอบของระบบ (components) 
    ที่ออกแบบและพัฒนาเสร็จแล้วกลับมาใช้ซ้ำในหลาย ๆ ส่วน เพื่อลดความซ้ำซ้อนในการเขียนโค้ดใหม่ ส่งผลให้สามารถลดระยะเวลาการทำงานและพัฒนาระบบได้เสร็จสิ้นตามเป้าหมาย
    \item การสูญหายของเอกสารหนังสือเชิญ ในขั้นตอนการติดต่อประสานงาน ทางโรงเรียนที่เป็นกลุ่มเป้าหมายได้ทำเอกสารหนังสือเชิญสูญหาย 
    ส่งผลให้กระบวนการติดต่อเพื่อขอลงพื้นที่เกิดความล่าช้า ทางผู้จัดทำจึงแก้ไขโดยการติดต่อประสานงานกับเจ้าหน้าที่ธุรการของภาควิชาวิศวกรรมคอมพิวเตอร์ 
    เพื่อขอออกสำเนาหนังสือเชิญอีกครั้ง และนำไปส่งมอบให้แก่ทางโรงเรียนใหม่เพื่อดำเนินการในขั้นต่อไป
\end{enumerate}

\section{\ifenglish Suggestions and further improvements \else ข้อเสนอแนะและแนวทางการพัฒนาต่อ\fi}

\subsection*{ด้านการออกแบบ UX/UI}
\begin{itemize}
    \item ควรมีการพัฒนาระบบเสียง โดยเพิ่มทั้งระบบเสียงบรรยาย (voice-over) ประกอบเนื้อหาบทเรียน เพื่อช่วยให้นักเรียนระดับประถมศึกษาเข้าใจเรื่องเศษส่วนได้ดียิ่งขึ้น 
    ควบคู่ไปกับการใช้ดนตรีประกอบ (background music) เพื่อดึงดูดความสนใจและสร้างบรรยากาศที่สนุกสนาน ทั้งนี้ ควรออกแบบให้ระบบเสียงทำงานสอดคล้องกัน เช่น 
    การลดระดับเสียงดนตรีประกอบลงโดยอัตโนมัติ (audio ducking หรือ fade) เมื่อมีเสียงบรรยาย เพื่อให้ผู้เรียนสามารถจดจ่อกับเนื้อหาได้อย่างเต็มที่โดยไม่ถูกรบกวน และเป็นการรองรับรูปแบบการรับรู้ที่หลากหลายของผู้เรียน
    ซึ่งข้อเสนอแนะนี้สามารถดำเนินการพัฒนาเพิ่มเติมได้ง่ายและรวดเร็ว เนื่องจากโครงสร้างของระบบในปัจจุบันได้ถูกออกแบบให้รองรับการทำงานในส่วนนี้อยู่แล้ว ผู้พัฒนาจึงเพียงแค่นำไฟล์เสียงเข้าไปติดตั้งในระบบก็สามารถใช้งานได้ทันที
    \item ควรพัฒนาระบบให้ผู้ใช้งานสามารถปรับแต่ง UI ได้ด้วยตนเอง เช่น การแต่งตัวมาสคอต, การเปลี่ยนภาพพื้นหลัง หรือการปรับเปลี่ยนสีธีม เพื่อเพิ่มความน่าสนใจ 
    กระตุ้นการมีส่วนร่วม และตอบสนองต่อความชอบส่วนบุคคลของนักเรียน 
    อย่างไรก็ตาม การพัฒนาระบบในส่วนนี้มีความซับซ้อนและต้องใช้ระยะเวลาในการทำงานค่อนข้างสูง เนื่องจากมีความจำเป็นต้องออกแบบและวาดภาพประกอบเพิ่มเติมสำหรับของตกแต่งใหม่ทั้งหมด 
    รวมถึงต้องวาดภาพ mascot ในลักษณะที่สวมใส่ของตกแต่งแต่ละชิ้นแยกกัน เพื่อให้ระบบสามารถนำไปประมวลผลและแสดงภาพได้อย่างสมบูรณ์
\end{itemize}

\subsection*{ด้านการพัฒนา Back-end}
\begin{itemize}
    \item ควรพัฒนาระบบการ synchronize ข้อมูลข้ามอุปกรณ์ เพื่อให้ผู้เรียนสามารถติดตามความคืบหน้าของบทเรียนได้อย่างต่อเนื่อง และสามารถเข้าถึงสื่อการสอนได้จากหลากหลายอุปกรณ์
    \item ควรพัฒนาระบบการจัดการ configuration ที่รองรับการแสดงผลลัพธ์แบบทันที (real-time preview) เพื่อให้ผู้พัฒนาสามารถตรวจสอบการเปลี่ยนแปลงรูปแบบหรือเนื้อหาของสื่อการสอนได้ควบคู่ไปกับการแก้ไข 
    ซึ่งจะช่วยลดข้อผิดพลาดและเพิ่มความสะดวกในการจัดการระบบ
\end{itemize}
